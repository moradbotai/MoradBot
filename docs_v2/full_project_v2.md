## 2. الهدف الأساسي (MVP)

MoradBot الإصدار الأول يركز على مشكلة واحدة فقط:

> أتمتة الردود على الأسئلة الشائعة داخل متاجر Salla باستخدام مساعد AI.

### الأهداف الرئيسية
- توفير 5 ساعات يومياً من الردود اليدوية
- توفير ردود آلية على مدار 24/7
- تقديم إجابات فورية مبنية على بيانات المتجر الموثّقة

لا منطق مبيعات. لا upselling. لا رسائل استباقية.
أتمتة FAQ فقط.

## 3. نطاق MVP

### 3.1 مدرج في MVP

#### أتمتة FAQ (5 فئات)
- توفر المنتج وسعره (عبر Salla Products API)
- مدة الشحن (FAQ يقدمه التاجر — فئة `shipping`)
- طرق الدفع (FAQ يقدمه التاجر — فئة `payment`)
- سياسة الإرجاع (FAQ يقدمه التاجر — فئة `returns`)
- الأسئلة العامة الأخرى (FAQ يقدمه التاجر — فئة `general`)

الفئات هي enum ثابت: `shipping`, `payment`, `returns`, `products`, `general`

#### تكامل Salla
- ربط OAuth (Custom Mode) — متجر واحد لكل تاجر
- صلاحية قراءة المنتجات فقط (`GET /products`)
- لا صلاحية الطلبات
- لا صلاحيات مالية

#### Chat Widget
- مضمّن داخل متجر Salla
- عربي فقط — البوت يرد دائماً بالعربية حتى لو كتب الزائر بالإنجليزية
- نص فقط (لا ملفات، لا صور، لا صوت)
- الرسالة الأولى تتضمن إفصاحاً واضحاً بأنه مساعد AI
- 7 حالات: `closed` → `open` → `typing` → `response` → `escalation` → `error` → `limit-reached`
- لا يظهر على مسارات `/checkout/*`
- لا cookies أو تخزين بدون موافقة صريحة من الزائر

#### لوحة تحكم التاجر (أساسية)
- عرض المحادثات (قراءة فقط)
- إحصائيات أساسية (عدد ردود البوت)
- زر On/Off
- عرض التصعيدات وإغلاقها يدوياً
- إدارة FAQ (إضافة / تعديل / حذف ناعم)
- معلومات الاشتراك

#### نظام التصعيد
- 3 محاولات توضيح فاشلة تُفعّل التصعيد
- جمع: وصف المشكلة + رقم الطلب (إن وجد) + طريقة التواصل
- التصعيد يظهر في لوحة التحكم
- التاجر يتواصل مع الزائر خارجياً
- التاجر يغلق التذكرة يدوياً

#### ذاكرة الجلسة
- ذاكرة الجلسة (داخل التذكرة)
- تتبع الزائر المستمر (عبر cookie — بعد موافقة صريحة فقط)
- البيانات الشخصية مشفرة (AES-256-GCM)

#### الموافقة والامتثال
- موافقة صريحة قبل تخزين أي بيانات شخصية للزائر
- معالجة البيانات بما يتوافق مع نظام حماية البيانات الشخصية (PDPL)
- سجلات تدقيق (audit logs) بفترة احتفاظ 90 يوماً

## 4. نموذج الأعمال

### هيكل الاشتراك
خطط شهرية ثابتة مبنية على عدد ردود البوت شهرياً:

| الخطة | الردود/شهر | السعر |
|-------|-----------|-------|
| Basic | 500 رد | 99 SAR |
| Mid | 2,000 رد | 299 SAR |
| Premium | 10,000 رد | 799 SAR |

### منطق الاستخدام
- ردود البوت فقط تُحسب ضمن الاستخدام
- رسائل الزائرين لا تُحسب
- التكلفة الداخلية تُحسب بالـ tokens (مخفية عن التاجر)

### إشعارات الحد
- عند 80% → إشعار للتاجر
- عند 100% → إشعار للتاجر
- لا رسوم تلقائية إضافية
- لا ترقية تلقائية

### سياسة الإلغاء
- تستمر الخدمة حتى نهاية دورة الفوترة
- لا إنهاء فوري

---

## 5. سياسة البيانات

### خلال فترة الاشتراك
- المحادثات مخزّنة
- البيانات الشخصية مشفرة (AES-256-GCM)
- تتبع الزائر المستمر عبر cookie — بعد موافقة صريحة فقط
- لا بيانات شخصية تُخزّن بدون موافقة صريحة من الزائر (PDPL)

### بعد الإلغاء
- فترة احتفاظ 30–90 يوماً
- ثم:
  - حذف جميع البيانات الشخصية
  - الاحتفاظ ببيانات مجهولة الهوية لتحسين النظام

---

## 6. العزل متعدد المستأجرين (Multi-Tenant Isolation)

- تسجيل دخول واحد لكل تاجر — متجر واحد لكل حساب في MVP
- تحمل تسامح صفري لتسريب البيانات
- أي تعرض للبيانات بين المتاجر = سبب مانع للإطلاق
- جميع الاستعلامات تتضمن `store_id` إلزامياً
- RLS مُفعّل على مستوى قاعدة البيانات (قاعدة: `auth.uid() = store_id`)

يمكن للموظفين الوصول للمحادثات لأغراض الدعم فقط، لكن:
- الحقول الحساسة مشفرة
- لا وصول نص عادي للبيانات الشخصية

## 7. البنية التقنية

### Stack المؤكد

| المكوّن | التقنية |
|--------|---------|
| API Runtime | Cloudflare Workers + Hono v4 (TypeScript) |
| قاعدة البيانات والمصادقة | Supabase PostgreSQL + RLS |
| مزود AI | OpenRouter → Gemini 2.0 Flash |
| نماذج Fallback | GPT-4 Mini، Claude 3.5 Sonnet |
| Chat Widget | Preact + Vite (هدف: <50KB gzipped) |
| لوحة التحكم | Next.js 15 App Router + Cloudflare Pages |
| مدير الحزم | pnpm + Turborepo |
| Linting/Formatting | Biome (tabs، double quotes، 100 char) |
| الاختبار | Vitest + Playwright |
| Rate Limiting | Cloudflare KV |
| التشفير | AES-256-GCM |
| البريد الإلكتروني | Resend |

### قاعدة البيانات (12 جدول)

`plans`, `stores`, `subscriptions`, `faq_entries`, `product_snapshots`, `visitor_sessions`, `tickets`, `messages`, `escalations`, `audit_logs`, `usage_tracking`, `bot_configurations`

### الاستضافة
- API: Cloudflare Workers
- لوحة التحكم: Cloudflare Pages
- قاعدة البيانات: Supabase (PostgreSQL)

### فلسفة البنية
- وكيل واحد (Single Agent) في MVP
- تعقيد أدنى للأشهر الستة الأولى

## القرارات المعمارية المحفوظة

- تسامح صفري لأي استعلام بدون `store_id` — خطأ فادح يوقف النظام
- OAuth Custom Mode فقط — لا Partner Mode في MVP
- الردود دائماً بالعربية — فرض على مستوى system prompt
- لا نشر تلقائي — كل نشر يمر بـ Pre-Deploy Checklist يدوياً
- الأسرار في Cloudflare Secrets فقط — لا في الكود أو `.env`

## ما تم تغييره من النسخة الأصلية

- "Target Merchant Size: 10-500 orders/month": تم التصحيح إلى "30–300 طلب/شهر" ليطابق PRD والـ CLAUDE.md
- قسم Infrastructure: تم استبدال الذكر المجرد لـ Cloudflare وSupabase بالـ stack التقني الكامل المؤكد
- قسم Business Model: تم استبدال المثال الوحيد (500 رد) بهيكل الخطط الثلاث الفعلي (Basic/Mid/Premium)
- قسم Branding: White-label مُعلَّم صراحةً كـ "مؤجل — ما بعد MVP"
- تمت إضافة قسم "حالة التنفيذ" يعكس Phases 1-3 المكتملة
- تمت إضافة اشتراط الموافقة الصريحة قبل تخزين البيانات الشخصية في قسم Data Policy
